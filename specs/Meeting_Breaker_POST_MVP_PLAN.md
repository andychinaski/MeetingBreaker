# Meeting Breaker — POST-MVP план развития

## 1. Назначение документа

Этот документ описывает развитие Meeting Breaker после завершения MVP.

Цели post-MVP:

- добавить новые типы встреч и переиспользуемые игровые механики;
- расширить главное меню;
- добавить три игровых режима;
- реализовать первый запуск и пошаговое обучение;
- добавить локальные таблицы лидеров;
- расширить настройки;
- добавить экран информации;
- улучшить линейное прохождение;
- сохранить возможность статического деплоя без backend.

## 2. Ограничения

На этом этапе:

- backend не используется;
- база данных не используется;
- онлайн-таблицы лидеров не используются;
- авторизация не используется;
- данные игрока и рекорды сохраняются в `localStorage`;
- приложение должно оставаться пригодным для деплоя на GitHub Pages;
- игровая логика не должна зависеть от React-компонентов;
- новые типы встреч не должны реализовываться отдельным классом для каждого типа.

# Часть I. Архитектура новых типов встреч

## 3. Цель

Расширить набор встреч так, чтобы они отличались не только цветом, названием и HP, но и поведением.

Основной принцип:

- используется один `MeetingBlock`;
- особенности встречи задаются набором поведений;
- параметры поведения задаются конфигурацией;
- поведение не должно напрямую изменять React;
- уничтоженный блок не должен выполнять отложенные действия.

## 4. Модель поведения

Добавить поддержку:

```ts
export interface MeetingBehavior {
  onCreate?(context: MeetingBehaviorContext): void;
  onHit?(context: MeetingBehaviorContext): void;
  onDestroy?(context: MeetingBehaviorContext): void;
  onUpdate?(context: MeetingBehaviorContext, delta: number): void;
  onBallCollision?(context: MeetingBehaviorContext): void;
  onNeighborDestroyed?(context: MeetingBehaviorContext): void;
  dispose?(): void;
}
```

Расширить тип встречи:

```ts
export interface MeetingType {
  id: string;
  title: string;
  category: MeetingCategory;

  colorToken: MeetingColorToken;
  maxHp: number;
  score: number;
  freedMinutes: number;
  dropChance: number;

  behaviorIds?: MeetingBehaviorId[];
  behaviorConfig?: Record<string, unknown>;
}
```

## 5. Базовые поведения

### 5.1. `moving`

Встреча перемещается внутри своей области.

Использование:

- Incident Call;
- Emergency Sync;
- Last-Minute Meeting.

Требования:

- не выходит за границы календарной колонки;
- физическое тело синхронизировано с отображением;
- движение корректно работает после паузы;
- таймеры и обработчики очищаются после уничтожения.

### 5.2. `blinking`

Встреча периодически меняет состояние:

- мигает;
- меняет цвет;
- становится полупрозрачной;
- предупреждает о смене фазы.

### 5.3. `recurring`

После уничтожения создаёт копию в другом свободном слоте.

Ограничения:

- число повторений ограничено;
- копии дают меньше очков;
- нельзя создавать бесконечный цикл;
- копия визуально помечается;
- копия не создаётся после завершения уровня.

### 5.4. `split`

После уничтожения создаёт несколько дочерних блоков.

Использование:

- Retrospective;
- Backlog Refinement;
- Task Decomposition.

Ограничения:

- дочерние блоки не наследуют `split`, если это явно не указано;
- дочерние блоки не пересекаются;
- число дочерних блоков ограничено.

### 5.5. `shielded`

Встреча не получает урон, пока не выполнено условие:

- не уничтожены связанные встречи;
- не завершена предыдущая фаза;
- существует защитный блок.

### 5.6. `linked`

Несколько встреч объединены через `groupId`.

Варианты:

- общий урон;
- уменьшенный урон, пока вся группа существует;
- уничтожение одной встречи повреждает остальные;
- уничтожение одной встречи снимает защиту с группы.

### 5.7. `accelerating`

Попадание по встрече увеличивает скорость шарика.

Используется для срочных встреч.

Скорость должна оставаться ниже глобального максимума.

### 5.8. `slow-field`

Встреча создаёт вокруг себя область замедления.

Скорость должна рассчитываться через модификаторы:

```ts
finalSpeed =
  baseSpeed *
  modeSpeedModifier *
  espressoModifier *
  meetingFieldModifier;
```

Не изменять скорость напрямую из разных систем.

### 5.9. `timed`

Встреча меняет состояние по таймеру:

- становится уязвимой;
- становится защищённой;
- начинает двигаться;
- создаёт другой блок;
- переключает фазу.

### 5.10. `bonus-drop`

Повышает вероятность выпадения указанного бонуса.

# Часть II. Новые типы встреч

## 6. Этап 1. Простые конфигурационные встречи

Добавить типы, которые используют существующие механики:

### Project Status

- HP: 2;
- средняя награда;
- стандартное поведение.

### Sprint Review

- HP: 2;
- длительность 60 минут;
- повышенное освобождённое время.

### Performance Review

- HP: 3;
- небольшой блок;
- высокий шанс Espresso Shot.

### Corporate Training

- HP: 3;
- крупный блок;
- высокий показатель освобождённого времени.

### Stakeholder Meeting

- HP: 4;
- крупный блок;
- высокая награда.

### Результат этапа

- встречи добавлены через конфигурацию;
- новые классы не создаются;
- тексты корректно помещаются;
- начисление очков и времени работает.

## 7. Этап 2. Incident Call

### Поведение

- перемещается в пределах своей колонки;
- мигает;
- при попадании ускоряет шарик;
- HP: 2;
- повышенный шанс бонуса;
- высокая награда.

### Проверить

- границы движения;
- работу после паузы;
- очистку таймеров;
- уничтожение движущегося блока;
- корректность физического тела.

## 8. Этап 3. Recurring Meeting

### Поведение

- после уничтожения появляется снова;
- копия создаётся в следующем доступном дне;
- максимум два повторения;
- копия даёт 50% награды;
- копия отмечается символом `↻`.

### Проверить

- отсутствие бесконечного цикла;
- корректное поведение в пятницу;
- поиск свободного слота;
- запрет создания после завершения игры;
- корректное начисление награды.

## 9. Этап 4. Retrospective

### Поведение

- HP: 2;
- после уничтожения создаёт 2–3 блока `Action Item`;
- дочерние блоки имеют 1 HP;
- дочерние блоки считаются обязательными;
- дочерние блоки не разделяются повторно.

### Проверить

- отсутствие пересечений;
- обязательность дочерних блоков;
- завершение уровня;
- однократное срабатывание.

## 10. Этап 5. Cross-Team Sync

### Поведение

- состоит из 2–3 связанных встреч;
- блоки имеют общий `groupId`;
- пока вся группа существует, урон снижен;
- уничтожение одного блока снимает защиту с остальных;
- между блоками отображается визуальная связь.

### Проверить

- работу группы;
- снятие защиты;
- очистку линий;
- уничтожение всей группы;
- отсутствие прямых ссылок между блоками.

## 11. Этап 6. Stakeholder Meeting с защитой

### Поведение

- основная встреча имеет 4 HP;
- изначально защищена;
- рядом расположены `Preparation Meeting`;
- после уничтожения подготовительных встреч защита снимается;
- основной блок становится уязвимым.

### Проверить

- защищённый блок не теряет HP;
- защита снимается один раз;
- попадание по защите не ломает комбо;
- награда начисляется один раз.

## 12. Этап 7. Corporate Training

### Поведение

- создаёт область замедления;
- шарик внутри области теряет скорость;
- после выхода скорость восстанавливается;
- после уничтожения встречи поле исчезает.

### Проверить

- вход и выход из поля;
- минимальную скорость;
- совместимость с Espresso Shot;
- удаление поля;
- отсутствие накопления модификаторов.

## 13. Этап 8. Go/No-Go Meeting

### Поведение

Состояния:

- `GO` — получает урон;
- `NO-GO` — отражает шарик без урона;
- `SWITCHING` — предупреждает о смене состояния.

Состояние меняется по таймеру.

### Проверить

- урон только в `GO`;
- остановку таймера на паузе;
- очистку таймера;
- визуальную понятность состояний;
- отсутствие сброса комбо при попадании в `NO-GO`.

## 14. Этап 9. Postmortem

### Поведение

- отсутствует в начале;
- появляется после уничтожения Incident Call;
- имеет 2 HP;
- становится обязательной встречей;
- гарантированно или с высоким шансом выдаёт Decline.

### Проверить

- корректный триггер появления;
- отсутствие преждевременного создания;
- поиск свободного слота;
- обязательность для завершения.

## 15. Этап 10. All Hands Boss

### Фаза 1

- 5 HP;
- статичный блок;
- стандартные столкновения.

### Фаза 2

После потери половины HP:

- начинает двигаться;
- создаёт два `Department Update`;
- получает временную защиту.

### Фаза 3

После уничтожения дочерних встреч:

- защита снимается;
- движение ускоряется;
- начинается финальная фаза.

### Победа

- усиленный эффект уничтожения;
- крупная награда;
- завершение уровня.

### Проверить

- однократные переходы фаз;
- очистку дочерних объектов;
- очистку таймеров;
- корректный restart;
- корректный выход в меню.

# Часть III. Главное меню и навигация

## 16. Структура главного меню

Главное меню должно содержать:

1. `Играть`
2. `Таблица лидеров`
3. `Настройки`
4. `Информация`

### Требования

- меню реализуется на React;
- Phaser создаётся только после запуска игровой сессии;
- возврат в меню полностью уничтожает игровую сцену;
- повторный запуск не создаёт второй canvas;
- выбранный пункт должен быть доступен мышью и клавиатурой.

# Часть IV. Первый запуск

## 17. Проверка имени игрока

При первом нажатии `Играть`:

1. Проверить наличие имени в `localStorage`.
2. Если имя не задано — показать поле ввода.
3. Провести валидацию.
4. Сохранить имя локально.
5. Разрешить изменение имени в настройках.

### Ограничения имени

- от 2 до 24 символов;
- пробелы по краям удаляются;
- пустое имя запрещено;
- HTML не должен интерпретироваться;
- имя отображается только как текст.

## 18. Предложение обучения

После имени:

- показать вопрос о прохождении обучения;
- варианты `Да` и `Нет`;
- добавить флажок `Больше не показывать`;
- выбор сохраняется локально;
- обучение можно запустить вручную из меню или информации.

## 19. Выбор управления

После выбора обучения:

- предложить `Клавиатура` или `Мышь`;
- добавить `Больше не показывать`;
- сохранить выбор;
- разрешить изменение в настройках.

### Модель данных

```ts
export interface PlayerPreferences {
  playerName: string | null;
  controlScheme: 'keyboard' | 'mouse';
  tutorialCompleted: boolean;
  skipTutorialPrompt: boolean;
  skipControlPrompt: boolean;
}
```

# Часть V. Пошаговое обучение

## 20. Общий принцип

Обучение должно быть отдельным сценарным уровнем.

Нельзя ограничиваться экраном с текстом.

Каждый следующий шаг открывается после выполнения действия игроком.

## 21. Последовательность обучения

1. Переместить платформу.
2. Запустить шарик.
3. Разбить обычную встречу.
4. Разбить встречу с несколькими HP.
5. Показать начисление свободного времени.
6. Показать комбо.
7. Потерять шарик и объяснить кофе.
8. Подобрать Espresso Shot.
9. Подобрать Decline.
10. Подобрать Async Mode.
11. Поставить игру на паузу.
12. Завершить учебную неделю.

## 22. TutorialController

Создать отдельный контроллер:

```ts
export interface TutorialStep {
  id: string;
  message: string;
  waitForEvent: GameEvent;
  highlightTarget?: string;
  allowSkip?: boolean;
}
```

### Требования

- tutorial не должен быть захардкожен в `GameScene`;
- шаги задаются конфигурацией;
- обучение можно пропустить;
- после прохождения сохраняется `tutorialCompleted`;
- повторный запуск доступен вручную;
- переход между шагами происходит один раз.

# Часть VI. Игровые режимы

## 23. Экран выбора режима

После первичного onboarding показать:

- `Прохождение`
- `Relax Mode`
- `Hard Mode`

Для каждого режима показать краткое описание правил.

## 24. Прохождение

Основной линейный режим:

- уровни открываются последовательно;
- вводятся новые типы встреч;
- сохраняется прогресс;
- у каждого уровня есть лучший результат;
- кофе работает;
- шарик постепенно ускоряется;
- шанс бонусов стандартный.

### Post-MVP развитие карты

Линейное прохождение расширяется с недели до месяца.

Месяц не является одним гигантским игровым полем.

Структура:

```text
Месяц
├── Неделя 1
├── Неделя 2
├── Неделя 3
├── Неделя 4
└── Финальная неделя / босс
```

Требования:

- каждая неделя остаётся отдельным уровнем;
- статистика суммируется за месяц;
- карта показывает прогресс;
- следующая неделя открывается после предыдущей;
- финальная неделя может содержать All Hands Boss.

## 25. Relax Mode

Бесконечный режим.

Правила:

- нет уровней;
- календарь периодически заполняется новыми встречами;
- шарик не ускоряется;
- скорость остаётся постоянной;
- кофе работает;
- шанс бонусов стандартный или повышенный;
- сложность растёт преимущественно через состав встреч;
- сессия продолжается до завершения кофе.

## 26. Hard Mode

Бесконечный режим.

Правила:

- нет уровней;
- календарь периодически заполняется новыми встречами;
- кофе отсутствует;
- потеря всех активных шариков завершает сессию;
- шарик постепенно ускоряется;
- бонусы выпадают реже;
- сложные типы встреч появляются раньше;
- результат должен иметь повышенный множитель.

Важно:

- отсутствие кофе реализуется отдельным правилом режима;
- нельзя использовать `coffeeCups: 0` как обычное состояние;
- HUD не должен показывать кофе;
- потеря основного и дополнительных шариков должна обрабатываться корректно.

## 27. Конфигурация режима

```ts
export interface GameModeConfig {
  id: 'campaign' | 'relax' | 'hard';

  endless: boolean;
  coffeeEnabled: boolean;
  initialCoffeeCups?: number;

  ballAccelerationEnabled: boolean;
  bonusDropMultiplier: number;
  scoreMultiplier: number;

  initialDifficulty: number;
  difficultyGrowth: number;
}
```

# Часть VII. Endless Meeting Generator

## 28. Назначение

Relax и Hard используют общий генератор встреч с разными настройками режима.

## 29. Логика генерации

1. Создать начальный набор встреч.
2. Следить за числом активных блоков.
3. Через заданный интервал запланировать новую волну.
4. Показывать предупреждение перед появлением.
5. Создавать встречи только в свободных слотах.
6. Увеличивать сложность со временем.
7. Завершать генерацию после проигрыша.

## 30. Волны

Каждая волна может увеличивать:

- число встреч;
- средний HP;
- долю специальных встреч;
- количество связанных групп;
- вероятность Incident Call;
- вероятность Recurring Meeting;
- скорость генерации.

## 31. Ограничения

- не более 100 блоков;
- не создавать блоки поверх существующих;
- не создавать новую волну после game over;
- не создавать больше двух сложных механик рядом;
- не допускать неразрешимой конфигурации;
- таймеры должны очищаться при restart и выходе.

# Часть VIII. Локальная таблица лидеров

## 32. Общий принцип

Таблица лидеров является локальной.

Данные хранятся в `localStorage`.

Рейтинг не синхронизируется между устройствами.

## 33. Разделение по режимам

### Прохождение

Хранить:

- лучший результат уровня;
- лучший результат недели;
- лучший результат месяца;
- общее освобождённое время.

### Relax Mode

Основной результат:

- освобождённое время;
- длительность сессии;
- уничтоженные встречи;
- максимальная волна.

### Hard Mode

Основной результат:

- очки;
- длительность сессии;
- максимальная волна;
- уничтоженные встречи.

## 34. Модель результата

```ts
export interface LeaderboardEntry {
  id: string;
  playerName: string;
  mode: GameModeId;

  score: number;
  freedMinutes: number;
  destroyedMeetings: number;
  maxCombo: number;
  wave?: number;
  durationSeconds: number;

  createdAt: string;
}
```

## 35. Требования

- отдельная вкладка для каждого режима;
- сортировка зависит от режима;
- хранить ограниченное число записей;
- поддержать очистку локальных результатов;
- корректно обрабатывать повреждённые данные;
- не называть локальный рейтинг глобальным.

# Часть IX. Настройки

## 36. Разделы настроек

### Управление

- клавиатура;
- мышь.

- музыка включена/выключена;
- громкость звуков;
- громкость музыки.

### Язык

Минимально:

- русский;
- английский.

### Внешний вид

- светлая тема;
- тёмная тема;
- цветовая схема блоков.

### Профиль

- имя игрока;
- повторный запуск обучения;
- сброс first-run подсказок.

## 37. Цветовые схемы блоков

Минимальные палитры:

- `default`;
- `pastel`;
- `high-contrast`.

Тема интерфейса и палитра блоков должны быть независимыми.

Пример:

```ts
export interface AppearanceSettings {
  uiTheme: 'light' | 'dark';
  meetingPalette: 'default' | 'pastel' | 'high-contrast';
}
```

В конфигурации встречи хранить `colorToken`, а не HEX-цвет.

## 38. Локализация

Все пользовательские тексты должны быть вынесены из компонентов и сцен.

Пример:

```ts
t('menu.play')
t('mode.relax.description')
t('tutorial.launchBall')
```

Требования:

- не хранить пользовательские строки внутри Phaser-логики;
- язык меняется без перезагрузки;
- длинные английские и русские строки не ломают интерфейс;
- настройки сохраняются.

# Часть XIII. На что обратить внимание

## 45. Риски

### Утечки таймеров

Новые типы встреч и endless-генератор используют таймеры.

Нужно очищать их при:

- уничтожении блока;
- restart;
- pause, если таймер должен останавливаться;
- выходе в меню;
- game over;
- завершении уровня.

### Конфликт модификаторов скорости

Нельзя напрямую менять скорость из разных бонусов и встреч.

Использовать одну систему итоговых модификаторов.

### Бесконечное создание блоков

Recurring и Split должны иметь жёсткие лимиты.

### Неразрешимые уровни

Shielded-блок не должен зависеть от уже уничтоженного или несуществующего блока.

### Hard Mode

Нельзя считать отсутствие кофе обычным нулевым значением.

Это отдельное правило режима.

### LocalStorage

Нужно:

- версионировать структуру данных;
- валидировать данные;
- использовать значения по умолчанию;
- уметь мигрировать старые настройки;
- не падать при повреждённых данных.

### Локализация

Нельзя использовать текст как ID игровой логики.

### Темы

Тема React и цвета Phaser должны обновляться согласованно.
